Hybrid Supabase mode: local speed, shared Postgres authority

The app started with two clear modes: local Drift/SQLite for one-device use, and client-server FastAPI/Postgres for a deployed backend. This proposal adds a third mode: Flutter plus local cache, with Supabase Postgres as the source of truth and RPC functions for financially critical writes.

One-Line Model

Supabase Postgres becomes the shared ledger. The phone database becomes a cache, draft store, and backup/export layer. RPC functions become the protected doorway for actions that must be atomic.

Keep

Drift/SQLite

Great for fast screens, offline reads, PDF generation input, recent state, and local backup. It should not independently invent official financial truth once multiple devices can write.

Add

Supabase Postgres

One shared source for invoices, products, customers, buyers, stock movements, and ledger transactions. Postgres gives transactions and constraints.

Constrain

Direct Table Writes

Reads can be broad. Critical writes should go through RPC so invoice numbers, inventory, and ledger rows are committed as one unit.

Architecture

Think of this as a third data mode beside the existing local and API modes. It borrows the local app's offline speed and the server mode's relational schema, but avoids running a separate FastAPI service.

Local Cache vs Postgres

The local database is still valuable. It just changes job title. It stops being the legal ledger and becomes the local working copy.

Reads are cache-first

Lists, detail pages, analytics, product picker, and PDFs can read from Drift. A background sync pulls updated rows from Supabase and refreshes Drift.

open screen read Drift immediately show "synced 2 minutes ago" pull changes from Supabase update Drift screen refreshes if anything changed

Critical writes are RPC-first

Creating invoices, recording collections, canceling invoices, and changing stock should call Supabase RPC. After success, the app updates Drift from the returned canonical rows or via the next sync pull.

tap "Create invoice" call rpc_create_invoice(request_id, draft_payload) Postgres commits official rows app receives invoice_id and invoice_number Drift cache stores the result

Offline starts conservative

For the first hybrid release, offline can mean read-only plus drafts. Avoid offline official invoices until an operation queue and conflict policy exist.

offline allowed: - browse last synced data - prepare invoice draft - export/share old PDFs offline blocked at first: - official invoice number - stock mutation - ledger mutation

Backup remains useful

The existing encrypted backup can still protect local state and provide a disaster-recovery export. Supabase should also have periodic SQL backups or exports once this becomes the authority.

device backup: encrypted Drift snapshot authority backup: Supabase SQL dump or managed backup/export rule: never restore a stale phone snapshot over newer Postgres truth

What RPC Means Here

RPC means "remote procedure call." In Supabase, the app calls a Postgres function as if it were an API endpoint. The function runs inside the database, close to the tables and constraints.

Why not write tables directly?

An invoice is not one row. It changes invoice headers, invoice items, stock movements, product quantities, and customer ledger rows. Direct table writes make partial failure and bad ordering easier.

Why RPC helps

One function validates the draft, locks the invoice counter, assigns the next number, inserts every related row, updates stock, writes ledger entries, and commits or rolls back as one transaction.

Flutter side:

final result = await supabase.rpc( 'create_invoice', params: { 'request_id': requestId, 'draft': draftJson, }, );

Postgres side, simplified:

create or replace function create_invoice(request_id uuid, draft jsonb) returns jsonb language plpgsql security definer set search_path = public as $$ declare next_number integer; begin -- 1. Return existing result when request_id was already processed. -- 2. Validate customer, products, quantities, GST mode, and totals. -- 3. Lock invoice counter row. -- 4. Insert invoice, items, stock movements, and ledger rows. -- 5. Return canonical invoice payload. end; $$;

Robustness and ACID Rules

This is the non-negotiable part. The architecture is only simple if the invariants are simple and enforced near the data.

Atomicity

Invoice header, items, stock movements, and ledger rows commit together or not at all.

Consistency

Constraints prevent duplicate invoice numbers, duplicate request IDs, invalid amounts, and impossible stock states where required.

Isolation

Concurrent devices do not choose invoice numbers independently. The database locks the counter or uses a sequence.

Durability

Once Supabase confirms success, the official result is in Postgres. Phones can rebuild their cache from the authority.

Idempotency

Every write keeps a request_id. If the network times out and the user retries, Postgres returns the first result instead of creating a duplicate.

RLS and RPC

Clients should read only allowed rows. Critical table writes can be denied directly and allowed through audited functions.

Migration Path From Today's App

Do not rewrite everything. Add a third data mode and move one business flow at a time. The existing service interfaces are a strength here.

Phase Build Done when
1. Schema Port backend/local tables to Supabase migrations. Add device_id, updated_at, deleted/archive markers, and sync cursor metadata where needed. Supabase can hold products, customers, buyers, profiles, invoices, items, stock movements, and transactions.
2. Auth and roles Use Supabase Auth or simple approved-user accounts. Add roles such as owner, operator, read_only. Only known family/business devices can access the project.
3. Read sync Implement pull-to-Drift for products/customers/buyers/invoices. Show last sync time. Any device can rebuild its local cache from Supabase.
4. RPC writes Implement create_invoice, cancel_invoice, record_collection, stock_adjustment, and catalog/profile upserts. Critical writes are transactionally correct with request_id replay.
5. Hybrid services Add HybridInvoicesService, HybridPaymentsService, HybridProductsService, and friends behind AppDependencies. The Flutter UI stays mostly unchanged.
6. Offline policy Start with offline read and drafts. Later add an operation queue only for operations with conflict-safe rules. The app is honest about what can be official while offline.

Cost and Scale Fit

Your stated scale is roughly 50 invoices per day, 2,000 products, 30 buyers, and 50 to 100 customers. That is small for Supabase Postgres.

Database size

Supabase Free currently lists a 500 MB database quota. A small invoice business should have a long runway if logs and old backups are controlled.

Users

Free tier currently lists 50,000 monthly active users. You need 5 to 10 known devices, not a public consumer app.

Operational cost

Money cost can be zero at first. The real cost is learning DB functions, access rules, migrations, and backup discipline.

For comparison, Firestore's free quota currently lists 1 GiB storage, 50,000 document reads per day, and 20,000 document writes per day. That also fits your volume, but it is less natural for a relational ledger.

Risks and Guardrails

Risk: business logic hidden in SQL

RPC is powerful, but PL/pgSQL can become hard to maintain. Keep functions small, test them, and document each contract.

Risk: false offline confidence

A phone can show cached data while stale. Use sync timestamps and block official writes when the app cannot reach Supabase in the first release.

Guardrail: append-only finance

Prefer reversals and adjustments over destructive edits. This makes audit, sync, and recovery dramatically easier.

Guardrail: export path

Keep the existing encrypted local backup and add Supabase SQL export habits. Free does not mean casual.
